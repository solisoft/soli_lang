Introduction

The Solilang MVC Framework provides a clean, organized structure for building web applications. It follows the Model-View-Controller pattern, separating concerns into logical components.

Type-Safe

Built with Solilang's type system for compile-time safety

Hot Reload

Instant updates during development without restarting

Scoped Middleware

Apply middleware to specific routes with fine-grained control

Template Engine

Powerful templating with layouts and dynamic content

Quick Start

Get your first MVC application running in minutes.

Create a New Project

bash
cargo run -- serve examples/mvc_app

Your application will be available at http://localhost:3000

⚠ The server supports hot reload! Edit any file and changes are reflected immediately.

Architecture

The MVC pattern separates your application into three main components:

Model

Handles data logic and business rules. Models interact with your database and return structured data to controllers.

View

Presents data to users. Views use templates to render HTML with dynamic content from controllers.

Controller

Orchestrates requests. Controllers receive input, interact with models, and return responses through views.

📖 This architecture promotes separation of concerns, making your code more maintainable and testable.

Project Structure

A typical MVC project has the following structure:

text
mvc_app/ ├── app/ │ ├── controllers/ # Route handlers │ │ └── home_controller.soli │ ├── middleware/ # Middleware functions │ │ ├── auth.soli │ │ └── cors.soli │ ├── models/ # Data models │ └── views/ # Templates ├── config/ │ └── routes.soli # Route definitions └── public/ # Static assets

Controllers

Controllers define the actions that handle HTTP requests. Each function in a controller corresponds to a route.

soli
// app/controllers/home_controller.soli fn index(req: Any) -> Any { return { "status": 200, "body": "Hello, World!" }; } fn about(req: Any) -> Any { return { "status": 200, "body": "About page" }; }

Views

Views handle the presentation layer, rendering HTML responses using ERB-like templates. Templates mix HTML with embedded Soli code.

Rendering Views from Controllers

Use the render() function to render a template with data:

soli
fn index(req: Any) -> Any { return render("home/index", { "title": "Welcome", "message": "Hello, World!", "users": [ {"name": "Alice", "email": "alice@example.com"}, {"name": "Bob", "email": "bob@example.com"} ] }); }

Template Structure

Templates are located in app/views/ and can use layouts for consistent page structure:

text
app/views/ ├── home/ │ ├── index.html # Home page template │ └── about.html # About page template ├── layouts/ │ └── application.html # Main layout with header/footer └── users/ ├── index.html # Users list └── show.html # User detail view

Template Syntax (ERB-like)

Mix HTML with embedded Soli code using special tags:

Syntax Description
<%= expr %> Output expression result
<% code %> Execute code (no output)
<% if cond %>...<% end %> Conditional block
<% for item in list %>...<% end %> Loop over collection

Example Template

erb
<!-- app/views/home/index.html --> <h1><%= title %></h1> <p><%= message %></p> <h2>Users</h2> <ul> <% for user in users %> <li><%= user["name"] %> - <%= user["email"] %></li> <% end %> </ul> <p>Total users: <%= count(users) %></p>

Conditionals

erb
<% if is_logged_in %> <p>Welcome back, <%= user_name %>!</p> <% else %> <p>Please log in.</p> <% end %> <% if count(users) > 0 %> <p>There are users.</p> <% end %>

Layouts

Layouts wrap content pages with common header, footer, and navigation:

erb
<!-- app/views/layouts/application.html --> <!DOCTYPE html> <html> <head> <title><%= yield "title" %></title> </head> <body> <header> <nav> <a href="/">Home</a> <a href="/about">About</a> </nav> </header> <main> <%= yield %> </main> <footer> &copy; 2024 My App </footer> </body> </html>
⚠ Use <%= yield %> in layouts to insert the content from child templates. Use <%= yield "title" %> to yield a specific section.

Middleware

Middleware functions intercept requests before they reach route handlers. They can modify requests, authenticate users, or short-circuit with responses.

Middleware Types

Type Option Description
Global-Only // global_only: true Runs for ALL routes, cannot be scoped
Scope-Only // scope_only: true Only runs when explicitly scoped to routes
Regular No option Runs globally by default, can also be scoped

Middleware Example

soli
// app/middleware/auth.soli // order: 20 // scope_only: true fn authenticate(req: Any) -> Any { let headers = req["headers"]; let api_key = headers["X-Api-Key"]; if (api_key == "secret-key-123") { return { "continue": true, "request": req }; } return { "continue": false, "response": { "status": 401, "body": "Unauthorized" } }; }

Scoped Middleware

Apply middleware to specific routes using the middleware() helper. This gives you fine-grained control over which routes require authentication, logging, or other middleware.

soli
// config/routes.soli // Apply authentication ONLY to admin routes middleware("authenticate", -> { get("/admin", "admin#index"); get("/admin/users", "admin#users"); }); // Public routes - no authentication required get("/", "home#index"); get("/about", "home#about");
⚠ Scoped middleware runs BEFORE global middleware, giving you precise control over request processing.

Routing

Define routes in config/routes.soli using the DSL helpers.

Basic Routes

soli
get("/", "home#index"); post("/users", "users#create"); put("/users/:id", "users#update"); delete("/users/:id", "users#destroy"); patch("/users/:id", "users#partial_update");

Namespaces

Group related routes under a common prefix.

soli
namespace("api", -> { middleware("authenticate", -> { get("/api/users", "users#index"); post("/api/users", "users#create"); }); });

Resources

Generate RESTful routes for a controller automatically using the resources() helper.

HTTP Method Path Action
GET/usersindex
GET/users/:idshow
GET/users/newnew
POST/userscreate
GET/users/:id/editedit
PUT/users/:idupdate
PATCH/users/:idupdate
DELETE/users/:iddestroy
soli
// Generate all RESTful routes for users resources("users", null);

Configuration

Middleware options are configured using special comments:

Option Example Description
// order: N // order: 10 Execution order (lower runs first)
// global_only: true // global_only: true Only runs globally, cannot be scoped
// scope_only: true // scope_only: true Only runs when explicitly scoped

CLI Commands

bash
# Start development server cargo run -- serve <project-path> # Parse routes without starting server cargo run -- parse <routes-file> # Run with hot reload (default) cargo run -- serve examples/mvc_app
